## 1. 你要做的是什么

**一句话**：一个 **后端 API 服务**（NestJS）+ **可选的管理界面**（Vue），实现：

- 用户注册 / 登录 / 改密 / 登出  
- 用 **JWT** 标识「谁登录了」  
- 用 **角色 + 权限** 控制「谁能调哪个接口」  

**小白理解**：  
- 后端像**餐厅后厨**：按菜单（接口）做菜（返回数据）。  
- 前端像**点餐界面**：顾客（管理员）点菜，后厨要认人、认权限，不能谁都能进冷库。

## 2. 先装什么软件

| 软件 | 作用 | 说明 |
|------|------|------|
| **Node.js**（建议 LTS 版本） | 运行 JavaScript/TypeScript 后端与前端构建 | 官网下载安装即可 |
| **pnpm** | 包管理器，装依赖比 npm 快 | `npm i -g pnpm` |
| **PostgreSQL** | 关系型数据库，存用户、角色、权限 | 本地装一个或用 Docker |
| **Git** | 版本管理 | 可选但强烈建议 |

**小白解释**：  
- **Node**：没有它，你写的 `.ts` 文件不能直接跑，需要 Node 来执行。  
- **pnpm**：在项目里执行 `pnpm install`，会下载 `package.json` 里列出的库（如 Nest、Prisma）。  
- **PostgreSQL**：数据存在磁盘上的「表格」里，比把数据写文件里正规、适合多人协作。

## 3. 10 分钟概念扫盲

### 3.1 HTTP 与接口

- 浏览器或前端用 **HTTP 请求**访问一个 **URL**，例如 `http://localhost:3000/auth/login`。  
- 常见方法：**GET**（查）、**POST**（提交/创建）、**PATCH**（部分更新）、**DELETE**（删）、**PUT**（整体替换）。

**小白**：把 URL 当成**门牌号**，GET/POST 当成**动作类型**。

### 3.2 JSON

- 接口之间传数据常用 **JSON** 格式：`{ "username": "admin", "password": "xxx" }`。  
- Nest 会自动把请求体解析成对象，把返回对象序列化成 JSON。

### 3.3 REST 风格（本项目的习惯）

- 资源名用复数路径，如 `/users`、`/roles`。  
- 用 HTTP 方法表达操作，而不是所有操作都叫 `/doSomething`。

### 3.4 JWT（JSON Web Token）

- 登录成功后，服务器签发一串**字符串**（Token），里面用密码学带上用户 id 等信息。  
- 以后请求在请求头里带上：`Authorization: Bearer <token>`，服务器验证签名和有效期，就知道你是谁。

**小白**：Token 像**当天有效的工牌**，不带工牌不让进某些门（受保护接口）。

### 3.5 Access Token 与 Refresh Token

- **Access Token**：短命，每次调业务接口都带着它。  
- **Refresh Token**：命长，只用来在 Access 过期时**换新 Access**（必要时也换新 Refresh）。  
- 本项目中 Refresh Token 还会**哈希存库**，防止被盗后无限续期。

### 3.6 ORM 与 Prisma

- **ORM**：用代码描述「表结构」和「增删改查」，少写原始 SQL。  
- **Prisma**：流行的 Node ORM，用 `schema.prisma` 定义模型，再生成类型安全的客户端。

## 4. 阶段一：空壳 Nest 项目跑起来

### 4.1 创建项目（若从零新建）

```bash
pnpm dlx @nestjs/cli new my-admin-api
cd my-admin-api
pnpm install
pnpm run start:dev
```

浏览器访问 `http://localhost:3000`（若默认有 Hello 路由）能看到响应即成功。

**解释**：`@nestjs/cli` 会生成标准目录：`src/main.ts`（入口）、`src/app.module.ts`（根模块）等。  
**模块（Module）**：Nest 用模块把功能分包，便于组织和注入依赖。

### 4.2 入口 `main.ts` 在干什么

典型会做三件事：

1. `NestFactory.create(AppModule)` 创建应用  
2. 注册**全局中间件/管道/过滤器**（验证、统一错误格式等）  
3. `listen(3000)` 监听端口  

**小白**：`main.ts` 是程序**启动的第一站**。

## 5. 阶段二：数据库与 Prisma

### 5.1 为什么用 Prisma

- 类型安全、迁移/同步方便、和 TypeScript 配合好。  
- 本仓库使用 **PostgreSQL**，连接串放在环境变量 `DATABASE_URL`。

### 5.2 安装与初始化（概念步骤）

```bash
pnpm add prisma @prisma/client @prisma/adapter-pg pg
pnpm exec prisma init
```

在 `prisma/schema.prisma` 里写 `User` 等模型，然后：

```bash
pnpm exec prisma generate   # 生成客户端代码
pnpm exec prisma db push    # 把 schema 推到数据库（开发常用）
```

**小白**：  
- `schema.prisma` 是**图纸**；  
- `db push` 是让数据库**按图纸建表**；  
- `generate` 是让项目里能写 `prisma.user.findMany()` 这类代码。

### 5.3 Nest 里怎么用 Prisma

写一个 `PrismaService`（继承 `PrismaClient`），在 `AppModule` 里 `providers` 注册，其它 Service **构造函数注入**即可。

**依赖注入（DI）**：你不用自己 `new PrismaService()`，Nest 帮你创建单例并传入——减少重复、方便测试。

本仓库若使用 **Prisma 7 + adapter-pg**，需在 `PrismaService` 里用 `PrismaPg` 传入 `DATABASE_URL`（见项目内 `src/prisma/prisma.service.ts`）。

## 6. 阶段三：统一返回格式与异常

### 6.1 为什么要统一

前端希望**所有成功响应**长这样，便于一层处理：

```json
{ "code": 200, "msg": "success", "data": { ... 实际数据 } }
```

### 6.2 实现思路

- **拦截器（Interceptor）**：在响应离开服务器前包一层 `{ code, msg, data }`。  
- **异常过滤器（Exception Filter）**：捕获错误，返回 `{ code: 状态码, msg: 错误信息, data: null }`。

**小白**：拦截器管「成功怎么包装」，过滤器管「出错怎么包装」。

## 7. 阶段四：用户与密码（注册登录）

### 7.1 用户表最少有什么

- `username`（唯一）、`passwordHash`（**不要存明文密码**）、`id`、`createdAt` 等。

### 7.2 密码为什么要哈希

用 **bcrypt** 等对密码做**单向哈希**，数据库泄露时攻击者难以还原原始密码。

注册流程：校验用户名不重复 → `hash(密码)` → 写入 `passwordHash`。  
登录流程：按用户名查出用户 → `compare(输入密码, passwordHash)`。

### 7.3 Controller / Service 分工

- **Controller**：只负责路由、参数装饰器、`DTO` 校验。  
- **Service**：业务逻辑、查库。

**DTO（Data Transfer Object）**：定义「接口期望收到什么字段」，配合 `class-validator` 做校验。

## 8. 阶段五：JWT 与刷新 Token

### 8.1 签发 Token

使用 `@nestjs/jwt` 的 `JwtService.signAsync`，payload 里至少包含：

- `sub`：用户 id  
- `username`  
- `type`：`'access'` 或 `'refresh'`（避免混用）

Access 与 Refresh 使用**不同密钥**（`JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET`），泄露一个不会立刻拖垮另一个。

### 8.2 保护路由

写一个 **Guard**：从请求头取出 `Bearer xxx`，用对应密钥 `verify`。  
验证失败抛 `UnauthorizedException`（401）。

在需要登录的接口上加：`@UseGuards(JwtAuthGuard)`。

### 8.3 刷新接口

客户端 POST 提交 `refreshToken`，服务端验证 refresh 类型、校验数据库里存的 refresh 哈希，通过后签发**新的** token 对，并更新库中 refresh 哈希。

### 8.4 环境变量里的过期时间

- `JWT_ACCESS_EXPIRES_IN`：纯数字表示**秒**（如 `900` = 15 分钟）。  
- **改 `.env` 后必须重启 Node 进程**；且已下发的旧 Token 不会变，需要**重新登录**才拿到新过期时间。

## 9. 阶段六：角色与权限数据模型

### 9.1 为什么要有「角色」和「权限」

- **角色**：如「运营」「财务」「超级管理员」，是一组权限的集合。  
- **权限**：细到「能否删除角色」「能否改用户角色」，用字符串编码如 `system:role:delete`。

### 9.2 常见表结构（与本项目一致）

- `User`  
- `Role`（`code` 唯一，如 `super_admin`）  
- `Permission`（`code` 唯一，如 `system:user:list`）  
- `UserRole`：用户与角色多对多  
- `RolePermission`：角色与权限多对多  

**小白**：用户不直接挂一堆权限，而是**挂角色**，角色再挂权限，方便批量管理。

## 10. 阶段七：RBAC 接口权限

### 10.1 目标

在「已登录」之上，再判断「有没有权限访问这个接口」。

### 10.2 实现步骤（概念）

1. 定义**权限码常量**（与数据库 `Permission.code` 一致）。  
2. 启动时或迁移时，把权限**同步进表**（upsert）。  
3. 写一个 `RbacService`：根据 `userId` 查角色 → 权限；若用户属于 `super_admin` 角色可**直接放行全部**。  
4. 写一个 `PermissionsGuard` + 装饰器 `@Authorize('system:role:list')`：  
   - 先 Jwt 认证（知道 userId）  
   - 再检查是否具备所列权限码（全部都要满足，AND）

### 10.3 为什么在 Controller 上挂装饰器

一眼能看出**哪个接口需要什么权限**，前后端也可共用同一套权限码字符串。

## 11. 阶段八：业务模块（用户列表、角色 CRUD）

### 11.1 角色模块

- `POST /roles` 创建、`GET /roles` 分页列表、`PATCH/DELETE` 更新删除。  
- `GET /roles/:id/permissions`、`PUT /roles/:id/permissions` 维护角色权限。

### 11.2 用户模块

- `GET /users` 列出用户及角色。  
- `PATCH /users/:id/roles` 给用户绑定角色（`roleIds` 数组，可清空）。

### 11.3 分页与校验

查询参数用 DTO + `class-transformer` 把 `page` 从字符串转成数字，并限制 `pageSize` 最大值，防止一次拉爆数据库。

## 12. 阶段九：前端 Vue 后台

### 12.1 技术选型

本仓库前端：**Vue 3 + Vite + Vue Router + Ant Design Vue**。

### 12.2 要接的关键点

1. **登录**：POST `/auth/login`，把返回的 `accessToken`、`refreshToken` 和用户信息（含 `permissionCodes`）存 `localStorage`。  
2. **请求头**：业务请求带 `Authorization: Bearer <accessToken>`。  
3. **401 处理**：Access 过期时，用 Refresh 调 `/auth/refresh-token` 换新，再重试原请求（本项目在 `frontend/src/api/client.ts` 中实现）。  
4. **菜单显示**：用 `permissionCodes` 或 `isSuperAdmin` 控制菜单项是否渲染。

### 12.3 跨域

开发时前端端口（如 5173）与后端（3000）不同，后端需 `app.enableCors()`。

## 13. 阶段十：种子数据与管理员

### 13.1 为什么需要 seed

第一次数据库是空的，没有权限数据、没有超级管理员，**谁都进不了管理功能**。  
用 `prisma/seed.ts`：

- 写入全部 `Permission`  
- 创建 `super_admin` 角色并绑定全部权限  
- 创建默认管理员用户并关联该角色  

执行：`pnpm db:seed`（具体脚本见 `package.json` 的 `prisma.seed`）。

### 13.2 默认账号（未改环境变量时）

- 用户名：`admin`  
- 密码：`Admin@123`  

可通过环境变量 `SEED_ADMIN_USERNAME`、`SEED_ADMIN_PASSWORD` 覆盖（以你项目 seed 脚本为准）。

## 16. 常见问题

### Q1：改了 `.env` 没效果？

先**完全重启**后端进程；JWT 过期时间只影响**新登录**签发的 Token，旧 Token 要**重新登录**才变。

### Q2：提示权限不足？

检查用户是否被分配了角色、角色是否包含对应权限码；超级管理员角色会跳过权限检查。

### Q3：Swagger 在哪里？

默认 `http://localhost:3000/api`，Bearer Token 可在 Swagger 里配置后调试接口。

### Q4：生产环境要注意什么？

- 修改所有默认密钥（JWT、数据库密码）  
- 关闭不必要日志、配置 HTTPS  
- 前端 `VITE_API_BASE` 指向正式 API 地址
